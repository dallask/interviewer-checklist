<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user input → store action | Session names (max 50 chars) enter store via renameSession / createSession |
| store → chrome.storage.local | Session data written via storageAdapter; manifest written via subscribe |
| chrome.storage.local → store | Session data read in switchSession / duplicateSession / deleteSession |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01-01 | Tampering | switchSession — cross-session write | mitigate | Call `storageAdapter.flushPending()` before any `set()`; all per-session fields + activeSessionId in ONE `set()` call (per SESS-04; verified by test) |
| T-06-01-02 | Tampering | deleteSession — data loss if undo buffer not captured | mitigate | Capture `undoBuffer` from storage read BEFORE `chrome.storage.local.remove()`; test asserts read precedes remove |
| T-06-01-03 | Tampering | session name injection via renameSession | accept | React JSX auto-escapes; no dangerouslySetInnerHTML; store accepts any string ≤50 chars; local-only extension, no backend |
| T-06-01-04 | DoS | Session proliferation via repeated createSession | accept | No UI affordance for automation; existing `#checkQuota()` in StorageAdapter warns at 80% usage; no additional guard needed |
| T-06-01-05 | Information Disclosure | Orphaned session:<id> key if deleteSession fails after remove | mitigate | `deleteSession` removes manifest entry in the same action sequence; storage key + manifest entry removed atomically within one async function |
| T-06-01-SC | Tampering | npm install | accept | No new packages introduced in Phase 6; all packages already audited in prior phases |
</threat_model>
