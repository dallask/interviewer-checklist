# Phase 6 — UI Design Contract

> Visual and interaction contract for Phase 6: Multiple Named Sessions & Switcher.
> Extends Phase 5 design system — all base tokens are inherited, not redefined.
> Generated by gsd-ui-researcher.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none | Chrome extension — components.json not applicable |
| Preset | not applicable | Manual Tailwind v4 utility classes throughout |
| Component library | none | Tailwind v4 utilities only; native `<dialog>` for all modals |
| Icon library | none | Text-character icon buttons (✎ ⧉ ×); no external icon library in Phase 6 |
| Font | system-ui, sans-serif | Browser default; inherited from Phase 4 |

**Styling approach:** Inherited from Phase 4–5. Tailwind v4 via `@tailwindcss/vite`. All styles as utility classes on JSX. `dark:` variant via `@custom-variant dark (&:where(.dark, .dark *))` in `src/app/styles.css`. No new CSS configuration required.

**Extension note:** Phase 6 adds one new trigger area in `ActionsGroup.tsx` (session label + switcher button), one new modal (`SessionSwitcherModal`), one new confirmation dialog (`DeleteSessionConfirmDialog`), and one new ephemeral notification (`UndoToast`). All follow patterns established in Phase 5.

## Spacing Scale

Inherited from Phase 4–5. No new tokens declared.

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `p-1` / `gap-1` | Icon button padding in session rows; gap between inline icon buttons |
| sm | 8px | `p-2` / `gap-2` | Session row vertical padding; gap between ActionsGroup elements |
| md | 16px | `p-4` / `gap-4` | Modal body padding; switcher modal content padding |
| lg | 24px | `p-6` / `gap-6` | Modal header/footer padding |
| xl | 32px | `p-8` / `gap-8` | Layout gaps (not used in Phase 6 new surfaces) |
| 2xl | 48px | `p-12` | Major section breaks (not used in Phase 6) |
| 3xl | 64px | `p-16` | Page-level spacing (not used in Phase 6) |

**Phase 6 exceptions:**

- Session row height: `min-h-[44px]` on each row — WCAG 2.5.8 touch target minimum for the clickable/tappable row area.
- Inline rename input: `w-full` within the row, same height as the row text (`py-1 px-1`); no separate sizing token.
- UndoToast: `fixed bottom-0 left-0 right-0` full-width; `py-2 px-4` internal padding — functional layout, not a spacing token.
- Session list max-height: `max-h-[352px]` (8 rows × 44px) with `overflow-y-auto` — functional constraint from CONTEXT.md (8 rows before scroll). Source: CONTEXT.md.
- Modal max-width: `max-w-md` (448px) — wider than ResetConfirmDialog (max-w-sm) but narrower than CandidateModal (max-w-lg); session list content does not need the extra width.

### ActionsGroup (extended from Phase 5)

Add two new elements at the top of the `ActionsGroup` content area (above "Expand all"), consistent with the session being the primary context:

**Active session label + switcher trigger:**

```
<!-- Session label (read-only, above the button) -->
<p class="text-xs font-normal text-gray-500 dark:text-gray-400 px-1 truncate"
   aria-label="Active session">
  {activeSessionName}
</p>

<!-- Switcher trigger button -->
<button
  type="button"
  id="open-session-switcher"
  onClick={() => sessionSwitcherRef.current?.showModal()}
```

- `UndoToast` is mounted at the root app level (inside `<App>`), not inside `ActionsGroup`. This allows it to overlay the full extension viewport when the switcher modal is closed. Source: CONTEXT.md (toast must not overlap the open modal).

### SessionRow (new sub-component)

A single row in the session list. Handles three states: default, active, and rename-in-progress.

**Default and active row:**

```
<li
  id="session-row-{session.id}"
  role="option"
  aria-selected="{isActive}"
  class="flex items-center gap-2 px-4 min-h-[44px]
         {isActive ? 'bg-blue-50 dark:bg-blue-900/20' : 'hover:bg-gray-50 dark:hover:bg-gray-800/50'}
         group"
>
  <!-- Active indicator (checkmark) or spacer -->
  <span class="w-4 shrink-0 text-xs {isActive ? 'text-blue-600 dark:text-blue-400' : 'text-transparent'}"
        aria-hidden="true">
    ✓
  </span>

  <!-- Session name (click to switch; click-to-edit rename on double-click or ✎ button) -->
  <button
    type="button"
    aria-label="Switch to {session.name}"
    class="flex-1 text-sm font-normal text-left text-gray-900 dark:text-gray-100 truncate
           focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none
           focus-visible:rounded"
    onClick={() => handleSwitchSession(session.id)}
  >
    {session.name}
  </button>

  <!-- Icon buttons (visible on row hover or focus-within) -->
  <div class="flex items-center gap-1 opacity-0 group-hover:opacity-100 group-focus-within:opacity-100 transition-opacity">
    <button
      type="button"
      aria-label="Rename {session.name}"
      class="p-1 text-xs text-gray-500 dark:text-gray-400 hover:text-gray-900 dark:hover:text-gray-100 rounded focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none"
      onClick={() => handleStartRename(session.id)}
    >
      ✎
    </button>
    <button
      type="button"
      aria-label="Duplicate {session.name}"
      class="p-1 text-xs text-gray-500 dark:text-gray-400 hover:text-gray-900 dark:hover:text-gray-100 rounded focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none"
      onClick={() => handleDuplicate(session.id)}
    >
      ⧉
    </button>
    <button
      type="button"
      aria-label="Delete {session.name}"
      class="p-1 text-xs text-red-600 dark:text-red-400 hover:text-red-800 dark:hover:text-red-300 rounded focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none"
      onClick={() => handleOpenDeleteConfirm(session.id)}
    >
      ×
    </button>
  </div>
</li>
```

- Icon buttons hidden at rest (`opacity-0`) and revealed on row hover or focus-within (`group-hover:opacity-100 group-focus-within:opacity-100`). `transition-opacity` provides a gentle fade — gated by `prefers-reduced-motion: reduce` (use `motion-safe:transition-opacity` variant). Source: UI-08 (Phase 4 established `prefers-reduced-motion` gate).
- Active checkmark is always rendered as `text-transparent` for inactive rows to maintain consistent column layout without layout shift.
- Session name button: full `flex-1` click target for switching, with `truncate` for overflow. `aria-label="Switch to {session.name}"` for screen readers.
- Delete icon button (×): red text, no red background — consistent with Phase 5 "Reset details" pattern (destructive intent signaled by color + label, no alarming background on the row itself).
- `role="option"` + `aria-selected` on each `<li>` so `role="listbox"` context is valid for screen readers.

**Rename-in-progress state:**

When the rename ✎ button is clicked (or the user triggers rename), the session name button is replaced by an inline input:

```
<li
  id="session-row-{session.id}"
  role="option"
  aria-selected="{isActive}"
  class="flex items-center gap-2 px-4 min-h-[44px]
         {isActive ? 'bg-blue-50 dark:bg-blue-900/20' : ''}"
>
  <!-- Active indicator spacer (same width) -->
  <span class="w-4 shrink-0" aria-hidden="true" />

  <!-- Inline rename input -->
  <input
    type="text"
    aria-label="Rename session"
    maxLength={50}
    defaultValue={session.name}
    class="flex-1 text-sm font-normal text-gray-900 dark:text-gray-100
           bg-white dark:bg-gray-800 border border-gray-300 dark:border-gray-600
           rounded px-1 py-1
           focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none"
    autoFocus
    onKeyDown={(e) => {
      if (e.key === 'Enter') commitRename(session.id, e.currentTarget.value);
      if (e.key === 'Escape') cancelRename();
    }}
    onBlur={(e) => commitRename(session.id, e.currentTarget.value)}
  />
</li>
```

- `autoFocus` moves focus to the input immediately when rename mode activates. Source: CONTEXT.md (inline click-to-edit — input replaces text on click, Enter/blur commits).
- `maxLength={50}` enforces the 50-character limit. Source: CONTEXT.md.
- On `Enter` or `blur`: validate non-empty → if empty, cancel rename and restore original name; if non-empty, commit to Zustand `renameSession(id, value)`.
- On `Escape`: cancel rename, restore original name, return focus to the row's name button.
- Icon buttons (✎ ⧉ ×) are NOT rendered while rename is in progress — the input replaces the entire name + icon area.

### UndoToast (new component)

Fixed-position ephemeral notification at the bottom of the extension viewport.

```
<div
  role="status"
  aria-live="polite"
  aria-atomic="true"
  class="fixed bottom-0 left-0 right-0 z-50
         bg-gray-900 dark:bg-gray-100
         text-gray-100 dark:text-gray-900
         flex items-center justify-between
         py-2 px-4
         motion-safe:animate-slide-up"
>
  <span class="text-xs font-normal">
    "{deletedSessionName}" deleted
  </span>
  <button
    type="button"
    onClick={handleUndo}
    class="text-xs font-normal text-blue-400 dark:text-blue-600
           hover:underline
           focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none
           ml-4 shrink-0"
  >
    Undo
  </button>
</div>
```

- `role="status"` + `aria-live="polite"` + `aria-atomic="true"` — screen readers announce the deletion result without interrupting in-progress reading. Source: WCAG 4.1.3 (Status Messages).
- `z-50` — renders above all sidebar and modal content.
- Inverted color scheme (dark background in light mode, light background in dark mode) — maximum contrast against the page.
- `motion-safe:animate-slide-up` — a 150ms `translateY(100%)→translateY(0)` slide animation, gated by `prefers-reduced-motion`. Define as a custom Tailwind animation in `styles.css`:
  ```css
  @keyframes slide-up {
    from { transform: translateY(100%); }
    to   { transform: translateY(0); }
  }
  ```
- Toast auto-dismisses after 10 seconds. Source: CONTEXT.md. Implemented via `setTimeout` in the component; dismissed immediately on "Undo" click or when the undo window expires.
- Toast appears after `DeleteSessionConfirmDialog` closes — never overlaps the open confirm dialog. The confirm dialog closes first, then the toast mounts. Source: CONTEXT.md specifics.
- When undo window expires: deletion is permanent. Closing the extension popup during the undo window also commits the deletion. Source: CONTEXT.md.
- "Undo" button restores the deleted session: calls `undoDeleteSession()` → re-inserts session data → removes toast.

## Interaction Contracts

### Open Session Switcher

1. User clicks "Switch session" in ActionsGroup.
2. `sessionSwitcherRef.current.showModal()` called → dialog renders with backdrop.
3. Focus moves to first focusable element: the close (×) button in the modal header.
4. Session list renders all sessions; active session has blue tint row + checkmark.
5. Icon buttons (✎ ⧉ ×) are hidden at rest; revealed on hover or keyboard focus-within.

### Switch Session

1. User clicks a session name button in the list.
2. If already active session: no-op (no state change, no close).
3. If different session:
   a. `flushPending()` called synchronously on `StorageAdapter` to commit any pending writes for the current session. Source: SESS-04.
   b. `switchSession(targetId)` dispatched to Zustand store.
   c. Store loads `session:<targetId>` from `chrome.storage.local` — scores, overrides, notes, topicNotes, customQuestions, candidate.
   d. Global state preserved: darkMode, sidebar groups, selectedSections, selectedDifficulties, hideMarked, searchQuery. Source: CONTEXT.md.
   e. Modal closes: `dialog.close()` → focus returns to `#open-session-switcher`.
4. Active session label in ActionsGroup updates to the new session's name.

### Create New Session

1. User clicks "New session" in the modal footer.
2. `createSession()` dispatched to Zustand.
3. New session created with name "Session N" (N = highest existing session number + 1). Source: CONTEXT.md.
4. `flushPending()` called for current session before switching.
5. New session becomes `activeSessionId`.
6. Modal closes → focus returns to `#open-session-switcher`.
7. ActionsGroup session label updates to "Session N".

### Rename Session (inline)

1. User clicks ✎ on a session row (or activates it via keyboard).
2. Session name button replaced by `<input>` with current name pre-populated, `autoFocus`.
3. User edits name (max 50 characters via `maxLength`).
4. `Enter` or `blur`: if non-empty → `renameSession(id, newName)` → Zustand updates manifest metadata → input replaced by name button displaying new name. If empty → cancel rename, restore original name.
5. `Escape`: cancel rename → restore original name → focus returns to ✎ button.
6. Rename does NOT switch the active session.

### Duplicate Session

1. User clicks ⧉ on a session row.
2. `duplicateSession(session.id)` dispatched. Source: CONTEXT.md specifics — duplicates the clicked session, not the active one.
3. New session created: name = "{originalName} (copy)", full state copied: scores, overrides, notes, topicNotes, customQuestions, candidate.
4. New session added to the list (appears at the bottom); does NOT become active automatically.
5. Modal remains open; list updates immediately to show the new entry.

### Delete Session (with confirm + undo)

1. User clicks × on a session row.
2. `DeleteSessionConfirmDialog` opens via `dialogRef.current.showModal()`. Session name shown in dialog body.
3. Focus moves to "Keep session" (safe default).
4. "Keep session" or `Escape`: dialog closes, no state change, focus returns to the × button.
5. "Delete" button clicked:
   a. `dialog.close()` (confirm dialog).
   b. `deleteSession(session.id)` dispatched immediately — storage write committed. Source: CONTEXT.md (deletion committed immediately on storage write).
   c. If deleted session was active: `switchSession()` auto-switches to most-recently-updated remaining session. If no sessions remain: `createSession("Session 1")` and switch. Source: CONTEXT.md.
   d. `SessionSwitcherModal` closes (both dialogs now closed). Focus returns to `#open-session-switcher`.
   e. `UndoToast` mounts at app root — ""{name}" deleted" + "Undo" button visible for 10 seconds.
6. "Undo" clicked within 10 seconds:
   a. `undoDeleteSession()` restores session data and metadata from in-memory undo buffer.
   b. If the deleted session was active at time of deletion: it becomes active again.
   c. Toast dismisses immediately.
7. 10 seconds elapse without undo: toast dismisses; undo buffer cleared; deletion is permanent.

### Close Modal

1. × button or `Escape` key in `SessionSwitcherModal`.
2. `dialog.close()` → backdrop removed.
3. Focus returns to `#open-session-switcher` (trigger button).

## Accessibility Contract

Inherits Phase 4–5 contract. Phase 6 additions:

| Requirement | Implementation | Source |
|-------------|----------------|--------|
| Session list semantics | `role="listbox"` on `<ul>`, `role="option"` + `aria-selected` on each `<li>` | WCAG 1.3.1 |
| Active descendant | `aria-activedescendant="session-row-{activeSessionId}"` on the listbox | WCAG 4.1.2 |
| Switcher modal ARIA | `<dialog aria-labelledby="session-switcher-title">` with `<h2>` inside | SESS-02 |
| Delete confirm modal ARIA | `<dialog aria-labelledby="delete-session-dialog-title">` with `<h2>` inside | SESS-03 |
| Modal focus traps | Manual Tab/Shift+Tab cycle within each `<dialog>`; `Escape` closes via native | SESS-02, SESS-03 |
| Modal focus restore | `dialog.addEventListener('close', () => triggerElement.focus())` on both dialogs | Phase 5 established pattern |
