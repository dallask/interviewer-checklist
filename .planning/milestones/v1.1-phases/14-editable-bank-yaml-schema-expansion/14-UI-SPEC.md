## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-text gap in add/remove trigger rows |
| sm | 8px | Inline form button gap (`gap-2`) |
| md | 16px | Default horizontal element padding (`px-4`) |
| lg | 24px | Inline form vertical padding (`py-3`) |
| xl | 32px | Left indent for topic-level content (`pl-8`) |
| 2xl | 48px | Left indent for question-level content (`pl-12`) |
| 3xl | 64px | Not used in this phase |

Exceptions:
- Touch targets for delete icon buttons on section/topic headers: minimum 44px tall via `min-h-[44px]` or existing `py-3` on the header row (matches existing QuestionCard score slider row pattern).
- "+ Add topic" trigger sits inside the TopicRow, inheriting `px-8 py-2` (32px left, 8px vertical) — mirrors the existing "+ Add question" button exactly.
- "+ Add section" trigger at ContentTree bottom uses `px-4 py-2` (16px left, 8px vertical) — same visual weight as "+ Add question".

Source: extracted from existing component classes in `TopicRow.tsx`, `CustomQuestionForm.tsx`, `QuestionCard.tsx`, `SectionRow.tsx`.

## Component Inventory

New components and modifications required:

### New: `AddSectionForm`

Inline form rendered below all sections at the bottom of ContentTree when the "+ Add section" trigger is activated.

Fields:
- Section name text input (required, `aria-label="Section name"`, placeholder "Section name…")
- Section icon input (optional, `aria-label="Section emoji icon"`, placeholder "🔧", single character)
- Submit button: "Add section" (blue-600 fill)
- Discard button: "Discard" (plain text, no fill)

Layout: `flex flex-col gap-3` wrapper with `px-4 py-3 bg-gray-50 dark:bg-gray-800/50 border-b border-gray-200 dark:border-gray-700`.

Trigger: "+ Add section" link-style button at ContentTree bottom — `text-xs text-gray-500 dark:text-gray-400 hover:text-gray-700 dark:hover:text-gray-200 px-4 py-2`.

Toggle: `addSectionOpen` boolean state in ContentTree. When form is open, trigger is replaced by the form (same `addQOpen` pattern in TopicRow).

### New: `AddTopicForm`

Inline form rendered below a section's last topic row when the "+ Add topic" trigger is activated.

Fields:
- Topic name text input (required, `aria-label="Topic name"`, placeholder "Topic name…")
- Optional description text input (`aria-label="Topic description"`, placeholder "Description (optional)…")
- Submit button: "Add topic" (blue-600 fill)
- Discard button: "Discard" (plain text)

Layout: identical to `CustomQuestionForm` — `px-8 py-3 bg-gray-50 dark:bg-gray-800/50 border-b border-gray-100 dark:border-gray-800 flex flex-col gap-3`.

Trigger: "+ Add topic" sits at the bottom of each section's rendered block, after all topics — `text-xs text-gray-500 dark:text-gray-400 hover:text-gray-700 dark:hover:text-gray-200 px-8 py-2`. Left indent `px-8` matches the existing "+ Add question" left indent.

Toggle: `addTopicOpen` boolean state per-section. One section can have its add-topic form open at a time.

### Modified: `SectionRow`

Add a `×` delete button visible only when `!row.isDefault`:

Placement: inside the existing `justify-between` flex row, after the question count span.

Classes: `text-xs text-red-600 dark:text-red-400 hover:text-red-800 dark:hover:text-red-300 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none ml-2 print:hidden` — identical to QuestionCard delete button.

Aria: `aria-label="Remove section {row.label}"`.

No confirmation dialog — immediate removal on click (mirrors custom question delete: no dialog, same destructive weight).

### Modified: `TopicRow`

Add a `×` delete button visible only when `!row.topic.isDefault`:

Placement: inside the topic header `button` element's `flex items-center gap-2` right span, after `TopicMarkDisplay`.

Classes: identical to section delete button — `text-xs text-red-600 dark:text-red-400 hover:text-red-800 dark:hover:text-red-300 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none ml-2 print:hidden`.

Aria: `aria-label="Remove topic {row.topic.name}"`.

Note: the delete button lives inside the `<button>` element that handles topic expand/collapse. Use `e.stopPropagation()` on the delete button click to prevent toggle firing — same fix applied in SCORE-07 for TopicMarkDisplay.

No confirmation dialog — immediate removal.

### Modified: `QuestionCard`

Extend delete button visibility from `row.isCustom === true` to also show for default questions (BANK-05).

The condition changes from `{row.isCustom === true && (...)}` to `{(row.isCustom === true || row.question.isDefault === true) && (...)}`.

For default questions, wire `onClick` to `removeDefaultQuestion(row.question.id)` instead of `deleteCustomQuestion`.

`aria-label` for default questions: `"Remove question"` (no "Delete custom question" label — default questions are not custom).

Visual appearance: identical to custom question delete — same red `×` character and same classes. No visual distinction between removing a default vs custom question from the user's perspective.

Source: `QuestionCard.tsx` lines 99–111 for existing pattern; CONTEXT.md D-04 for the extension spec.

## Phase-Specific Constraints

1. **ContentTree virtualizer**: ContentTree uses `@tanstack/react-virtual`. The "+ Add section" form and trigger must be added as proper virtual rows (type: `add-section-trigger` and `add-section-form`) in `buildFlatRows` output so the virtualizer can measure and position them correctly. Do not render them outside the virtualizer container. Estimated height: trigger row ~32px, form row ~120px.

2. **TopicRow `×` button inside `<button>`**: The topic header is a `<button>`. Nesting a `<button>` inside a `<button>` is invalid HTML. The delete icon for user-added topics must be a `<button>` placed adjacent to (not inside) the topic header button, or the topic header must be refactored to a `<div role="button">`. The recommended approach: render delete as a sibling `<button>` in a wrapping `<div className="flex">` around both, removing the outer `<button>` and making the expand area a `<div role="button" tabIndex={0}>`. If TopicRow refactor is too large, use `stopPropagation` and accept the nesting violation with a comment noting the deviation.

3. **Default question key scheme**: QuestionCard derives `questionId` as `` `${row.topicId}-q${row.index}` `` (index-based). The `removeDefaultQuestion` action receives `question.id` (the V4Question id field from the bank). These are different identifiers. BANK-05 store action must accept the V4Question `id`, not the score key. The score key is for score lookup only.

4. **Print suppression**: All new add/remove affordances carry `print:hidden`. No new elements are visible in print output.

5. **Dark mode**: All new interactive elements must carry both light and dark color classes — no light-only styling. Pattern: `text-{color}-{shade} dark:text-{color}-{dark-shade}`.
