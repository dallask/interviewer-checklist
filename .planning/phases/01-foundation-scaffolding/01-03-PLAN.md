    - Test 2: If a .js file in dist/ contains "eval(", the script exits 1 and stderr contains "FAIL [MV3 CSP]"
    - Test 3: If a .js file in dist/ contains "localhost", the script exits 1 and stderr contains "FAIL [dev artifact]"
    - Test 4: If dist/manifest.json permissions is not exactly ["storage"], the script exits 1 and stderr contains "FAIL [permissions]"
    - Test 5: If dist/manifest.json action.default_popup is present, the script exits 1 and stderr contains "FAIL [manifest]"
  </behavior>

  <action>
    Create scripts/ directory and write scripts/check-dist.js as a Node ES module (the package.json "type": "module" makes all .js files ES modules by default).

    The script must:
    1. Import readdirSync and readFileSync from 'node:fs', join from 'node:path' (use node: prefix for all Node built-ins).

    2. Define FORBIDDEN_PATTERNS array with these four regexes: /eval\(/, /new Function\(/, /unsafe-eval/, /<script[^>]*>[^<]/
       These catch direct eval, indirect eval, CSP directive strings, and inline script content.

    3. Define DEV_PATTERNS array with these five regexes: /localhost/, /127\.0\.0\.1/, /vite-hmr/, /@vite\/client/, /5173/
       These catch CRXJS issue #860 dev-server artifact leakage.

    4. Implement checkFile(filepath) — readFileSync the file as utf-8, iterate FORBIDDEN_PATTERNS and DEV_PATTERNS, console.error and process.exit(1) on first match.

    5. Implement walk(dir) — readdirSync with withFileTypes:true, recurse into directories, call checkFile on .js and .html files.

    6. Implement the manifest checks (before walk):
       - Parse dist/manifest.json
       - Assert JSON.stringify(manifest.permissions) === '["storage"]' — exit 1 with "FAIL [permissions]" message if not
       - Assert !manifest.action?.default_popup — exit 1 with "FAIL [manifest]: action.default_popup is present" if present

    7. Call walk('dist') then console.log('All dist/ safety checks passed.')

    The script uses process.exit(1) for failures (not throw) so the exit code is deterministic.
    The script uses the 'dist' path as a relative string — it must be run from the repo root (which is how package.json invokes it via "node scripts/check-dist.js").

    After writing the file, validate:
    - Run `npm run build && npm run ci:check-dist` — both must exit 0
    - Run a negative test: create a temp file dist/evil.js containing "eval('x')", run the script, verify it exits non-zero, then delete the temp file.
  </action>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| dist/ → CI safety check | check-dist.js is the last gate before zip upload; must catch all MV3 violations |
| GH Actions secrets → CWS API | OAuth secrets must be stored in GH Actions secrets, never in workflow YAML or source |
| semver tag → release trigger | Any tag matching v*.*.* triggers the release; tag must be signed or at minimum from a trusted branch |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-01 | Tampering | dist/ contents (eval/localhost/bad-perms) | mitigate | scripts/check-dist.js greps every .js and .html file in dist/ and exits 1 on any match; both CI workflows call this before zip/upload |
| T-03-02 | Information Disclosure | CWS OAuth credentials in workflow YAML | mitigate | Credentials stored as GH Actions secrets (${{ secrets.CWS_* }}); EXTENSION_ID stored as a variable (not secret — it is public on the store); never hardcoded in YAML |
| T-03-03 | Elevation of Privilege | release.yml triggered by arbitrary tag push | accept | Repo is private/personal; tag creation requires push access; CWS API will reject uploads with invalid credentials; acceptable risk for a solo developer project |
| T-03-04 | Denial of Service | key.pem committed to git | mitigate | .gitignore contains *.pem and key.pem (Plan 01); CWS manages the key after first upload; documented in PATTERNS.md |
| T-03-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan; chrome-webstore-upload-cli is already in devDependencies from Plan 01; used via npx in the workflow (no new install) |
</threat_model>
