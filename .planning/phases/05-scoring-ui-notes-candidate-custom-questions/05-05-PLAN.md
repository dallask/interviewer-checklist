<objective>
Fix the latent data-loss bug in bootstrap.ts identified as CR-03 in the phase 5 code review.

Root cause: Scenario B (valid V2 manifest present) reads each session key and validates it with `V2SessionSchema`. Phase 5 writes sessions as V3 format (`version: 3`). V3 sessions fail the `V2SessionSchema` literal version check, causing `sessionParseResult.success = false`, and `createDefaultSession(s.id)` is returned — silently replacing all scores, notes, and custom questions with an empty default. The main.tsx workaround (reading the session key directly, bypassing bootstrap's `sessions` return value) hides this in the happy path, but any future consumer of `bootstrap().sessions` would see empty state.

Fix: In the Scenario B session-loading loop, try V3SessionSchema first. If it passes, use the V3 output. If it fails, try V2SessionSchema. If that also fails, fall back to `createDefaultSession(s.id)`.

Purpose: Eliminate the latent data-loss path so that `bootstrap()` correctly returns V3 sessions for all future consumers (Phase 6 session switcher, Phase 7 YAML export, etc.).
Output: Updated src/storage/bootstrap.ts with version-aware session validation; new tests in src/storage/bootstrap.test.ts covering the V3 round-trip.
</objective>
