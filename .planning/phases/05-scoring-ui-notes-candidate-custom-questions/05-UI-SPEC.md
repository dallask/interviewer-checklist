# Phase 5 — UI Design Contract

> Visual and interaction contract for Phase 5: Scoring UI, Notes, Candidate & Custom Questions.
> Extends Phase 4 design system — base tokens are inherited, not redefined.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none | No components.json; Chrome extension — not applicable |
| Preset | not applicable | Manual Tailwind v4 utility classes throughout |
| Component library | none | Tailwind v4 utilities only; `<dialog>` native element for modals |
| Icon library | none in Phase 5 | Text labels and × characters; icon library deferred to Phase 9 |
| Font | system-ui, sans-serif | Browser default; inherited from Phase 4 |

**Styling approach:** Inherited from Phase 4. Tailwind v4 via `@tailwindcss/vite`. All styles as utility classes on JSX. `dark:` variant via `@custom-variant dark (&:where(.dark, .dark *))` in `styles.css`. No new CSS configuration required.

**Extension note:** This phase extends `QuestionCard.tsx` and `TopicRow.tsx` from Phase 4 with interactive scoring controls. New standalone components are `ScoreSlider`, `TopicMarkDisplay`, `NotesTextarea`, `CandidateModal`, `ResetConfirmDialog`, and `CustomQuestionForm`.

## Spacing Scale

Inherited from Phase 4. No new tokens declared. Exceptions added for this phase:

| Token | Value | Tailwind Class | Usage |
|-------|-------|----------------|-------|
| xs | 4px | `p-1` / `gap-1` | Icon gaps, tight inline padding, badge padding |
| sm | 8px | `p-2` / `gap-2` | Compact element spacing, slider track margins |
| md | 16px | `p-4` / `gap-4` | Default element spacing, modal body padding, notes textarea padding |
| lg | 24px | `p-6` / `gap-6` | Modal header/footer padding, section padding |
| xl | 32px | `p-8` / `gap-8` | Layout gaps |
| 2xl | 48px | `p-12` | Major section breaks (not used in Phase 5) |
| 3xl | 64px | `p-16` | Page-level spacing (not used in Phase 5) |

**Phase 5 exceptions:**
- Score slider thumb touch target: minimum 44px tall container (`min-h-[44px]`) — WCAG 2.5.8. Source: CONTEXT.md.
- Override input width: fixed `w-16` (64px) — functional constraint to align with computed mark display.
- Notes textarea min-height: `min-h-[80px]` (80px) — readable minimum for free-text entry; not an 8-point token but the nearest functional minimum.
- Modal max-width: `max-w-lg` (512px) — functional constraint for candidate details 6-field form.
- Dialog backdrop: full-screen `fixed inset-0` — not a spacing token.

### New for Phase 5: Mark Band Color Tokens

Mark band colors map to `getMarkBand()` return values from `src/scoring/index.ts`. Applied to mark displays in `TopicRow`, `SectionFilter`, and `TopicMarkDisplay`.

| Band | Return value | Light token | Dark token | Tailwind (light) | Tailwind (dark) | Trigger condition |
|------|-------------|-------------|------------|------------------|-----------------|-------------------|
| none | `"none"` | `text-gray-400` | `dark:text-gray-500` | `text-gray-400` | `dark:text-gray-500` | No questions scored yet |
| low | `"low"` | `text-red-500` | `dark:text-red-400` | `text-red-500` | `dark:text-red-400` | Mark 0–3.9 |
| mid | `"mid"` | `text-yellow-500` | `dark:text-yellow-400` | `text-yellow-500` | `dark:text-yellow-400` | Mark 4–5.9 |
| good | `"good"` | `text-green-600` | `dark:text-green-400` | `text-green-600` | `dark:text-green-400` | Mark 6–7.9 |
| high | `"high"` | `text-emerald-600` | `dark:text-emerald-400` | `text-emerald-600` | `dark:text-emerald-400` | Mark 8–10 |

Source: 05-CONTEXT.md decisions. The `none` band uses gray-400 (lighter than text-muted gray-500) to distinguish "not yet scored" from "scored zero."

**Note:** Phase 4 defined provisional band colors (excellent/good/marginal/poor + orange). Phase 5 replaces those with the five-band contract above matching `getMarkBand()` return values. The Phase 4 placeholders showed "—" in `text-gray-400` — that behavior is preserved for `band === "none"` with the same token.

### TopicMarkDisplay (new component)

Displays the computed mark and optionally an override input side by side.

**Layout:**

```
<div class="flex items-center gap-1" role="group" aria-label="Mark for {topicName}">
  <!-- Computed mark (colored by band) -->
  <span class="text-xs tabular-nums {bandColorClass}">
    {override !== null ? override.toFixed(1) : computedMark !== null ? computedMark.toFixed(1) : "—"}
  </span>

  <!-- Override input (always rendered, visually minimal) -->
  <input
    type="number"
    min="0"
    max="10"
    step="0.1"
    aria-label="Override mark for {topicName}"
    value="{override ?? ''}"
    placeholder="override"
    class="w-16 text-xs text-gray-700 dark:text-gray-300 bg-transparent border border-gray-300 dark:border-gray-600 rounded px-1 py-0.5 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none placeholder:text-gray-300 dark:placeholder:text-gray-700"
  />

  <!-- Clear override button (visible only when override is set) -->
  {override !== null && (
    <button
      type="button"
      aria-label="Clear override mark for {topicName}"
      class="text-xs text-gray-400 hover:text-gray-600 dark:hover:text-gray-200 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none"
    >
      ×
    </button>
  )}
</div>
```

- Computed mark shown using `bandColorClass` from the mark band color tokens table above.
- Override input `type="number"` `step="0.1"` allows fine-grained override. On blur: parse → clamp to [0, 10] → update Zustand `overrides` record. Source: CONTEXT.md decision.
- Clear button (×) sets `overrides[topicId] = null` → reverts to computed mark. Source: CONTEXT.md decision.
- `role="group"` + `aria-label` groups the computed mark and override input semantically for screen readers.
- `tabular-nums` on mark value prevents layout shift as number changes.
- Override input `w-16` (64px) fixed width to align display across topic rows.

## Interaction Contracts

### Score Slider

1. User moves slider (mouse drag, keyboard arrow keys, touch).
2. `onChange` fires → dispatch `setScore(questionId, value)` to Zustand immediately (no debounce — live recompute needed).
3. Zustand recomputes `computeTopicMark` for parent topic via `computeTopicMark()` from `src/scoring/index.ts`.
4. `TopicMarkDisplay` re-renders with new computed mark + updated band color.
5. Section filter mark labels re-render with updated `computeSectionMark` and `computeOverallMark` values.
6. StorageAdapter receives the updated session state via the existing Zustand subscribe → storageAdapter.write (300ms debounce). Source: Phase 4 established pattern.
7. Score display updates to `{value} / 10`.

### Override Input

1. User types a number in the override input (0–10).
2. On `blur`: parse value. If empty string → `setOverride(topicId, null)` (clear override). If valid number in [0, 10] → `setOverride(topicId, clamped)`. If invalid (NaN or out of range) → reset input to current override or empty.
3. Computed mark display updates to use override value if set; reverts to computed mark if cleared.
4. Band color re-evaluates based on active display value (override takes precedence).
5. Clear (×) button: `setOverride(topicId, null)` → mark reverts to `computeTopicMark`.

### Notes Textarea (question + topic)

1. User opens notes panel via "Add notes" / "Add topic notes" button.
2. `aria-expanded` updates to `true` on toggle button; textarea becomes visible.
3. User types in textarea → `onChange` fires → debounced 300ms → `setNote(noteId, value)` dispatched to Zustand.
4. On Zustand write, StorageAdapter persists updated `notes` / `topicNotes` records.
5. On session reload, notes values restored from StorageAdapter → textarea pre-populated.
6. "Hide notes" collapses textarea. Notes value persists regardless of collapsed state.

### Candidate Modal

1. User clicks "Candidate details" in Actions group.
2. `dialog.showModal()` called → dialog becomes visible with backdrop.
3. Focus moves to first focusable field (Name input).
4. Tab cycles through: Name → Email → Role → Date → Interviewer → Details → Reset details → Discard changes → Save details → (wraps to Name).
5. Shift+Tab reverses cycle.
6. Escape: `dialog.close()` (native) → focus returns to "Candidate details" trigger button.
7. "Discard changes" button: `dialog.close()` → no state change → focus returns to trigger.
8. "Save details" button: read form values → dispatch `setCandidate(candidateDetails)` to Zustand → `dialog.close()` → focus returns to trigger.
9. Reset details button: clear all form inputs → dispatch `setCandidate(null)` to Zustand. Dialog remains open.
10. If candidate already saved: pre-populate all fields from Zustand `candidate` record on `dialog.showModal()`.

### Reset Confirm Dialog

1. User clicks "Reset all" in Actions group.
2. `dialog.showModal()` called.
3. Focus moves to "Keep scores" button (safe default — not the destructive action).
4. Tab cycles: Keep scores → Reset → (wraps to Keep scores).
5. Escape: `dialog.close()` → no state change → focus returns to "Reset all" trigger.
6. "Keep scores": `dialog.close()` → no state change.
7. Reset: call `storageAdapter.snapshot()` → dispatch `resetAll()` to Zustand (clears scores, overrides, notes, topicNotes, customQuestions, candidate, filters) → `dialog.close()` → focus returns to trigger.

### Custom Question Add

1. User clicks "+ Add question" in a topic block.
2. `CustomQuestionForm` renders inline below the topic notes area.
3. Focus moves to question text input.
4. User fills question text, selects difficulty, clicks "Add question".
5. On submit (and non-empty text): `addCustomQuestion({ id: custom-${topicId}-${Date.now()}, topicId, text, level })` → Zustand update → question appears in topic list with "custom" badge and a difficulty pill matching the selected level.
6. "Discard question" button: collapse form, no state change.
7. Custom question participates in scoring (included in `computeTopicMark`), filtering, and future YAML export. Source: SCORE-05.

### Custom Question Delete

1. User clicks × on a custom question card.
2. Immediate: `deleteCustomQuestion(questionId)` → Zustand update.
3. No confirmation dialog. Source: CONTEXT.md decision.
4. Topic mark recomputes immediately.

### Section Filter Mark Display (Phase 5 activation)

Phase 4 showed "—" in `text-gray-400`. Phase 5 activates live marks:

1. Each section filter entry displays computed section mark using `computeSectionMark`.
2. Mark rendered as `{mark.toFixed(1)}` with band color class from the mark band tokens table.
3. If no questions in the section have been scored: display "—" in `text-gray-400` (band = "none").
