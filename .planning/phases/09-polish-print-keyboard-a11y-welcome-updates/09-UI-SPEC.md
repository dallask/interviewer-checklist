## Spacing Scale

Declared values (must be multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon gaps, inline badge padding, dismiss button gap |
| sm | 8px | Button internal padding (py-2), list item gap, sidebar gap-2 |
| md | 16px | Section padding (px-4), modal body padding |
| lg | 24px | Modal internal spacing (mb-6), welcome page section gap |
| xl | 32px | Welcome page hero vertical padding |
| 2xl | 48px | Welcome page between-section breaks |
| 3xl | 64px | Welcome page top/bottom page padding |

Exceptions:
- Update banner vertical padding: 8px top + 8px bottom (py-2) — matches UndoToast pattern
- Dismiss button touch target: minimum 44px height on welcome page CTAs
- Welcome page CTA buttons: py-4 px-6 (16px / 24px) for prominent tap target — 32px vertical padding plus text height exceeds 44px touch target minimum

Source: ActionsGroup.tsx (px-3 py-2), UndoToast.tsx (py-2 px-4), ResetConfirmDialog.tsx (p-6), StorageToast.tsx (px-4 py-3).

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Body | 14px (text-sm) | 400 (font-normal) | 1.5 |
| Label | 12px (text-xs) | 400 (font-normal) | 1.5 |
| Heading | 16px (text-base) | 600 (font-semibold) | 1.2 |
| Display | 20px (text-xl) | 600 (font-semibold) | 1.2 |

Notes:
- Body (14px/400) is the established app default, used in all existing buttons and paragraph text — source: ResetConfirmDialog.tsx (`text-sm font-normal`), ActionsGroup.tsx (`text-sm`).
- Label (12px/400) used for metadata text — source: ActionsGroup.tsx active session name (`text-xs font-normal`), UndoToast.tsx (`text-xs font-normal`). To reinforce the 2px size distinction from body (14px), label text must also use `text-gray-400 dark:text-gray-500` color differentiation, not body's `text-gray-700 dark:text-gray-300`.
- Heading (16px/600) used for modal titles — source: ResetConfirmDialog.tsx (`text-base font-semibold`).
- Display (20px/600) used for welcome page section headings only. Default — no existing precedent at this size, but consistent with 4-step type scale.

**Welcome page only — extended scale:**
The welcome.html page is a standalone full-page surface; it may use a wider type scale.

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Page title | 28px (text-3xl) | 600 (font-semibold) | 1.2 |
| Section heading | 20px (text-xl) | 600 (font-semibold) | 1.2 |
| Body paragraph | 16px (text-base) | 400 (font-normal) | 1.6 |
| Caption / step label | 12px (text-xs) | 400 (font-normal) | 1.5 |

The wider scale (up to 28px) is permitted on welcome.html only because it is a full-page tab, not a constrained sidebar panel. The 2-weight rule (400 + 600) is maintained throughout.

## Color

All colors use Tailwind v4 utility classes. Dark-mode handled via `.dark` class variant declared in styles.css (`@custom-variant dark (&:where(.dark, .dark *))`).

| Role | Light value | Dark value | Usage |
|------|------------|------------|-------|
| Dominant (60%) | white / gray-50 | gray-900 | Page background, dialog background, welcome page background |
| Secondary (30%) | gray-100 | gray-800 | Sidebar background, button backgrounds, card surfaces |
| Accent (10%) | blue-500 | blue-400 | Focus rings only (see reserved-for list) |
| Info / update | amber-50 border-amber-300 text-amber-800 | yellow-900/30 border-yellow-700 text-yellow-200 | Update banner background |
| Destructive | red-600 | red-500 | Destructive action buttons (Reset only — not new in phase 9) |

Accent reserved for:
- `focus-visible:ring-2 focus-visible:ring-blue-500` on all interactive elements (keyboard focus ring only)
- No other use of blue in the app shell

Source: ResetConfirmDialog.tsx (focus-visible:ring-blue-500), StorageToast.tsx (yellow warning palette), ActionsGroup.tsx (red-600 destructive button), Sidebar.tsx (gray-100/gray-800 sidebar).

**Update banner color decision:** Use amber/yellow warning palette (`bg-amber-50 border border-amber-300 text-amber-800` / dark: `bg-yellow-900/30 border-yellow-700 text-yellow-200`) — matches StorageToast.tsx warning pattern exactly. This is informational, not destructive. Source: CONTEXT.md ("top sticky banner"), StorageToast.tsx (established warning color palette).

**Welcome page color:** Same dominant/secondary palette. No new colors introduced. CTA "Start demo session" button uses `bg-blue-600 text-white hover:bg-blue-700` — the only place in the extension where a filled blue button appears (welcome page is a marketing/onboarding surface, not the app shell).

## Surface-by-Surface Contract

### 1. Welcome Page (welcome.html)

A standalone full-page tab. React or static HTML — Claude's discretion based on Vite config (vite.config.ts has a single entry; add `welcome: 'public/welcome.html'` if static, or a second Vite entry if React).

**Layout:**
- Max-width container: `max-w-2xl mx-auto px-6 py-16` (640px content width centered on full page)
- Vertical flow: page title → pin nudge section → two audience flow sections → CTA

**Page title:**
- Text: "Interviewer Checklist"
- Style: `text-3xl font-semibold text-gray-900 dark:text-gray-100 mb-2`

**Subtitle:**
- Text: "A structured interview scoring tool — browser-based, private, no account needed."
- Style: `text-base font-normal text-gray-600 dark:text-gray-400 mb-8`

**Pin-to-toolbar nudge section:**
- Heading: "Pin to your toolbar" (`text-xl font-semibold mb-3`)
- Body: "Click the puzzle-piece icon in Chrome's toolbar, find Interviewer Checklist, then click the pin icon." (`text-base font-normal text-gray-600`)
- Visual: a static SVG illustration or `<img>` showing the puzzle-piece → extension → pin flow. Width: `w-full max-w-xs` centered or left-aligned. Alt text: "Chrome extensions menu showing the pin button next to Interviewer Checklist"

**Audience flow sections (two cards, side-by-side on md+ or stacked on mobile):**
- Layout: `grid grid-cols-1 md:grid-cols-2 gap-6 my-8`
- Each card: `bg-gray-50 dark:bg-gray-800 rounded-lg p-6 border border-gray-200 dark:border-gray-700`
- Card 1 heading: "For interviewers" (`text-xl font-semibold mb-2`)
- Card 1 body: "Run structured technical interviews. Score questions 0–10 with difficulty weighting, capture notes, and generate an AI feedback prompt." (`text-base font-normal text-gray-600 dark:text-gray-400`)
- Card 2 heading: "For candidates" (`text-xl font-semibold mb-2`)
- Card 2 body: "Follow along during your interview. See which areas are being assessed and review the scoring rubric afterward." (`text-base font-normal text-gray-600 dark:text-gray-400`)

**CTA:**
- Primary button — "Open the extension" (opens the extension tab via `chrome.tabs.create`):
  `bg-blue-600 text-white text-base font-semibold px-6 py-4 rounded hover:bg-blue-700 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none`
- Secondary link — "View demo session" (switches to the seeded demo session already in storage):
  `text-blue-600 dark:text-blue-400 text-base font-normal underline` below or beside the primary CTA

**Dark mode:** Welcome page inherits the OS preference (`prefers-color-scheme: dark`) via Tailwind's dark mode. No toggle on the welcome page (app shell controls dark mode preference).

### 2. Update Banner (POLISH-07)

Position: sticky top banner inside the main content area, above the question list. NOT a toast. NOT a new tab.

**Markup pattern:**
```
<div role="status" aria-live="polite" aria-atomic="true"
     class="sticky top-0 z-30 flex items-center justify-between
            bg-amber-50 dark:bg-yellow-900/30
            border-b border-amber-300 dark:border-yellow-700
            text-amber-800 dark:text-yellow-200
            px-4 py-2 text-sm font-normal">
  <span>New in v{version}: <button class="underline font-normal">What's new</button></span>
  <button aria-label="Dismiss update banner" class="...">×</button>
</div>
```

**Copy:**
- Banner text: "Updated to v{version}."
- "What's new" inline button (opens CHANGELOG collapsible in sidebar footer)
- Dismiss button: `aria-label="Dismiss update banner"` with `×` glyph

**Color:** amber/yellow warning palette — matches StorageToast.tsx `bg-yellow-50 border-yellow-300 text-yellow-800` (light) / `bg-yellow-900/30 border-yellow-700 text-yellow-200` (dark). Source: CONTEXT.md decision + StorageToast pattern.

**Dismiss behavior:** clicking `×` sets `dismissedUpdateVersion` in `chrome.storage.local` to current version; banner disappears and does not return for this version.

**"What's new" behavior:** clicking inline "What's new" link scrolls sidebar to and opens the CHANGELOG collapsible in the sidebar footer.

### 3. Keyboard Shortcut Indicator (POLISH-02, POLISH-03)

No persistent visual affordance in the app shell (no keyboard shortcut cheatsheet panel). Shortcuts are:
- `/` → focus sidebar search input
- `\` → toggle sidebar open/closed
- `Esc` → clear search / close active modal

**Tooltip hint (optional, Claude's discretion):** The search input placeholder may hint `/` — `placeholder="Search questions… (/)"`. This is a zero-cost discoverability aid. No other shortcut indicators are added to the shell.

**Focus indicator after `/`:** When `/` fires and focuses the search input, the existing `focus-visible:ring-2 focus-visible:ring-blue-500` ring provides the visual feedback. No additional animation needed.

**`_execute_action` manifest command:** Declared in `manifest.json` under the `commands` key. No UI element is needed; this is an OS-level shortcut registration.

### 4. Print Layout (POLISH-05)

Implemented via Tailwind `print:` variants in component JSX. No separate CSS file.

**Elements to hide on print (add `print:hidden` class):**
- `<aside>` (sidebar) — `print:hidden`
- Update banner — `print:hidden`
- View toolbar (Expand all / Collapse all / Hide marked / Dark mode buttons) — `print:hidden`
- Score sliders (`<input type="range">`) — `print:hidden`
- All `<button>` elements that are action-only (Reset, Switch session, AI prompt, Candidate details) — `print:hidden`
- StorageToast / UndoToast — `print:hidden`
- Custom question form inputs — `print:hidden`

**Elements to show on print (override collapsed/hidden via `print:` variants):**
- All collapsed topic rows: `print:block` overrides collapsed hide — use `data-collapsed` attribute selector via `print:[&[data-collapsed]]:block` or add `print:block` to the hidden class that collapses them
- All question card bodies: same pattern — `print:block`
- Notes textareas: `print:resize-none print:overflow-visible print:h-auto` to auto-expand

**Print-specific layout additions:**
- Candidate name/role header at top of print: `print:block hidden` wrapper containing `<h1>{candidate.name}</h1><p>{candidate.role} — {candidate.date}</p>` — hidden on screen, shown on print
- Section/topic headings: existing headings are already visible; no change needed
- Scores: displayed as plain text (the mark band display already renders text) — sliders hidden, numeric mark visible

**Page breaks:** `print:break-inside-avoid` on each topic row to prevent awkward mid-topic page breaks.

### 5. Sidebar Footer — Version Display + CHANGELOG (POLISH-06)

The sidebar's last `<SidebarGroup>` is "Actions" (ActionsGroup.tsx). Below the Actions group, add a footer area directly in `<aside>` (Sidebar.tsx), after the last `<SidebarGroup>`.

**Version row:**
```
<div class="mt-auto px-3 py-2 text-xs font-normal text-gray-400 dark:text-gray-500
            border-t border-gray-200 dark:border-gray-700 print:hidden">
  v{version}
  <button class="ml-2 underline text-xs font-normal text-gray-400 hover:text-gray-600
                 dark:text-gray-500 dark:hover:text-gray-300
                 focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none">
    What's new
  </button>
</div>
```

**CHANGELOG collapsible:**
- Placed immediately below the version row, inside a `<details>` element or controlled via local state `[changelogOpen, setChangelogOpen]`
- Trigger: the "What's new" button in the version row (also triggered by the update banner "What's new" link)
- Container: `max-h-64 overflow-y-auto text-xs font-normal text-gray-600 dark:text-gray-400 px-3 pb-3`
- Content: CHANGELOG.md rendered as plain preformatted text or simple paragraph-per-version-block layout — Claude's discretion on markdown rendering depth
- Print: `print:hidden` on the CHANGELOG container

**`mt-auto` placement:** The sidebar is `flex flex-col`; using `mt-auto` on the footer pushes it to the bottom of the sidebar regardless of sidebar height. Source: Sidebar.tsx (`flex flex-col`).

### 6. Focus Trap Audit Visual Contract (POLISH-04)

No new visual elements. The audit confirms compliance with the existing pattern established in ResetConfirmDialog.tsx:

**Required pattern for all 5 modals:**
- `dialogEl.querySelectorAll<HTMLElement>('button, input, textarea, select, [tabindex]:not([tabindex="-1"])')` — focusable element selector
- Tab wraps from last to first; Shift+Tab wraps from first to last
- `close` event fires `document.getElementById('{trigger-id}')?.focus()` to restore trigger focus
- `aria-labelledby` pointing to modal `<h2>` heading ID

| Condition | Behavior |
|-----------|----------|

| Focus in app shell (no modal, no input) | All shortcuts active |

## Accessibility Contract

All elements in this phase follow the existing app pattern:

- All interactive elements: real `<button>` or `<a>` — no `div onClick`
- Focus rings: `focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none` on every interactive element
- Update banner: `role="status" aria-live="polite" aria-atomic="true"` (non-disruptive announcement)
- Welcome page: semantic HTML — `<main>`, `<h1>` for page title, `<h2>` for section headings, `<section>` per audience card
- Pin illustration: descriptive `alt` text (see Surface Contract section 1)
- Keyboard shortcut hook: uses `document.addEventListener('keydown', ...)` inside a `useEffect` with cleanup; guard checks `document.activeElement`
- Print candidate header: `aria-hidden="true"` or `hidden` attribute on screen (screen reader already sees candidate data in the modal)

Source: UI-07 (REQUIREMENTS.md), existing component patterns.
