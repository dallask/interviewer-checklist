<task type="checkpoint:human-action" gate="blocking-human">
  <name>Task 1: User publishes PRIVACY.md to a stable HTTPS URL and records it (CWS-01 hosting, per D-09)</name>
  <what-built>
Plan 10-01 produced `PRIVACY.md` at the repo root with full CWS-01 coverage (local-only storage, no transmission, no analytics, removal on uninstall). The document is ready to publish as-is.
  </what-built>
  <how-to-verify>
1. Pick a hosting destination. Any stable HTTPS URL works; the CWS reviewer must be able to load it without auth. Common options (pick whichever you control):
   - GitHub Pages: push PRIVACY.md to a `docs/` directory or `gh-pages` branch on the project's repo, enable Pages in repo settings → resulting URL is typically `https://<owner>.github.io/<repo>/PRIVACY` or similar. (Free, stable, recommended.)
   - GitHub raw rendered file: NOT recommended — raw.githubusercontent.com is not a "stable hosting URL" by CWS conventions even though it is accessible. Prefer Pages.
   - Personal site / Netlify / Vercel / Cloudflare Pages: upload the rendered markdown or HTML version.
   - GitHub repo file URL (`https://github.com/<owner>/<repo>/blob/main/PRIVACY.md`): acceptable as a fallback if the repo is public.

2. Publish. The exact steps depend on the host you chose:
   - For GitHub Pages: in repo Settings → Pages, set Source to `main` branch and `/docs` (or `/(root)`), wait for the green checkmark, then visit the URL to confirm the page renders.
   - For other hosts: follow that host's deployment flow.

3. Open the live URL in a fresh browser tab (incognito to confirm public accessibility). Verify the page renders the full privacy policy text and that the URL begins with `https://`.

4. Open `docs/cws-submission.md`. Find the section "Privacy policy URL" with the placeholder `https://<user-publishes-this-in-plan-10-03>`. Replace the placeholder with the actual live URL you just confirmed.

5. Commit the change: `git add docs/cws-submission.md && git commit -m "docs(10): record live privacy policy URL"`.

6. Type `done <url>` where `<url>` is the live URL you published. If you cannot publish for any reason, describe the blocker instead.
  </how-to-verify>
  <resume-signal>Type `done <https-url>` once the privacy policy is live at the URL and docs/cws-submission.md has been updated and committed. Or describe a blocker.</resume-signal>
</task>

<task type="checkpoint:human-action" gate="blocking-human">
  <name>Task 2: User uploads dist.zip and submits to CWS (CWS-02, CWS-05, per D-09)</name>
  <what-built>
- Plan 10-02 produced `dist.zip` (ready for upload) and three 1280x800 screenshots in `cws-assets/screenshots/`.
- Plan 10-01 + Task 1 above populated `docs/cws-submission.md` with every dashboard field's exact text including the live privacy policy URL.
- The user holds the active Chrome Web Store developer account credentials (out of orchestrator scope).
  </what-built>
  <how-to-verify>
1. Sign in to the Chrome Web Store Developer Dashboard: https://chrome.google.com/webstore/devconsole/

2. If this is a new listing, click "New item" and upload `dist.zip` from the repo root. Wait for the upload + manifest parse to complete. If the dashboard rejects the zip for any reason (invalid manifest, oversize asset, etc.), stop and surface the error — do not proceed.

3. Open `docs/cws-submission.md` in your editor and keep it visible alongside the dashboard.

4. Store listing tab — populate exactly these fields by copy-paste from docs/cws-submission.md:
   - Title: "Interviewer Checklist" (or the canonical title from manifest.json `name`).
   - Short description (paste from "Short description" section).
   - Detailed description (paste from "Detailed description" section).
   - Category: choose Developer Tools or Productivity per the recommendation in cws-submission.md section "Category and language".
   - Language: English.

5. Graphics assets — upload all three PNGs from `cws-assets/screenshots/`:
   - `01-scoring-view-populated.png`
   - `02-sidebar-with-filters.png`
   - `03-ai-prompt-modal.png`
   The CWS dashboard accepts 1280x800; if any upload is rejected for size, regenerate that screenshot per CWS-SCREENSHOTS.md before proceeding. Promotional images (440x280, 920x680, 1400x560) are out of scope per 10-CONTEXT.md deferred ideas — skip.

6. Privacy practices tab — populate:
   - Single purpose description: paste from "Single-purpose description" section.
   - Permission justifications:
     - `storage`: paste from "Permissions justification — `storage`" section (the exact phrasing from CWS-02).
     - No other permissions to justify.
   - Host permission justification: paste from "Host permissions justification" section.
   - Remote code: paste from "Remote code justification" section.
   - Data usage declarations: answer each toggle exactly as listed in the "Data usage declarations" table in cws-submission.md. PII (name, email): yes collected, no transmitted, no sold, no shared. All other categories: no.
   - Three required certifications: check all three boxes (no transfer to third parties, no use unrelated to stated purpose, no use for creditworthiness / lending).
   - Privacy policy URL: paste the live URL recorded in cws-submission.md by Task 1.

7. Distribution tab — confirm visibility "Public", regions: default (all), pricing: free.

8. Review all tabs for completeness. The dashboard will flag any missing required fields with a red indicator. Resolve every flag before submitting.

9. Click "Submit for review". The dashboard should change status to "Pending review" or equivalent. Take a screenshot of the dashboard status indicator and save it to `cws-assets/screenshots/04-submission-status.png` for your records (this fourth screenshot is for repo evidence only, not for the CWS listing).

10. Commit the status screenshot: `git add -f cws-assets/screenshots/04-submission-status.png && git commit -m "docs(10): record CWS submission status"`.

11. Type `submitted` once the dashboard shows the submission is in review. If the dashboard rejects the submission at click time, describe the rejection reason instead so the orchestrator can plan a fix iteration.

Note: per 10-CONTEXT.md decision D-09, Phase 10 is considered COMPLETE when the submission enters review (not when the listing goes live). Final CWS acceptance (listing live) is recorded manually by the user later in STATE.md outside this phase.
  </how-to-verify>
  <resume-signal>Type `submitted` once the CWS dashboard shows the listing in review status and the status screenshot is committed. Or describe a dashboard error / rejection.</resume-signal>
</task>
