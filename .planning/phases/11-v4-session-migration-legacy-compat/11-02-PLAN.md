    Replace the existing session loop body (lines 126-141) with the new V4 migration loop from PATTERNS.md §bootstrap.ts pattern:

    1. Declare `const sessions: Record<string, V4Session> = {}` and `const failedSessionIds: string[] = []`.
    2. For each session in manifest.sessions:
       a. Try v.safeParse(V4SessionSchema, rawSession). If success → sessions[s.id] = result.output; continue. (Pitfall 3 guard: already-V4 sessions skip migration.)
       b. Try v.safeParse(V3SessionSchema, rawSession). If success:
          - Write pre-v4 snapshot: `await chrome.storage.local.set({ [\`snapshot:${s.id}:pre-v4-${Date.now()}\`]: rawSession })` directly (NOT via storageAdapter or adapter.snapshot — Pitfall 5). Per D-05.
          - Run `try { const v4 = migrateV3ToV4(v3Result.output); const validV4 = v.safeParse(V4SessionSchema, v4); if (validV4.success) { await chrome.storage.local.set({ [key]: v4 }); sessions[s.id] = validV4.output; } else { console.error('[bootstrap] V4 validation failed for session:', s.id, validV4.issues); failedSessionIds.push(s.id); } } catch (err) { console.error('[bootstrap] migrateV3ToV4 threw for session:', s.id, err); failedSessionIds.push(s.id); }` — per D-06.
          - continue.
       c. Try v.safeParse(V2SessionSchema, rawSession). If success → treat as V2 that missed migration. Run runMigrations and then migrateV3ToV4 chain, or simply assign createDefaultV4Session(s.id). The V2→V3→V4 chain is: run migrateV2ToV3 then migrateV3ToV4. Alternatively for defense-in-depth, just: sessions[s.id] = createDefaultV4Session(s.id) (V2 sessions should have been migrated by v1.0 bootstrap already; a V2 surviving to v1.1 is a corrupted session).
       d. All other cases (unknown/corrupt) → `sessions[s.id] = createDefaultV4Session(s.id)`. Do NOT push to failedSessionIds here — there is no V3 blob to protect.

    3. Return `{ manifest, sessions, failedSessionIds }`.

    CRITICAL ANTI-PATTERNS:
    - Do NOT use `adapter.snapshot()` for the pre-v4 write — it calls #trimSnapshots which deletes older snapshots
    - Do NOT include failed sessions in the returned sessions map (per RESEARCH.md Pitfall 2)
    - Do NOT call migrateV3ToV4 on a V4 session (the V4 guard at step 2a must fire first)
    - The V3SessionSchema parse must occur before writing the pre-v4 snapshot (parse confirms it is a valid V3 before writing backup)

    Update the scenario C block (v1 migration at lines 71-101) to also return failedSessionIds: [] in its success path. Same for the Scenario D recovery path.
