    Step 1 — Update the import on line 2 of ContentTree.tsx:
    Change `import { useRef, useState } from 'react'` to `import { useEffect, useRef, useState } from 'react'`.

    Step 2 — Insert scroll-after-add logic after the `rowVirtualizer` declaration block (after line 45, before the `const virtualItems` line). Add:
    - A `useRef<number>` named `prevRowsLengthRef` initialized to `rows.length` to track previous row count
    - A `useEffect` that runs when `rows` changes; inside:
      - If `rows.length > prevRowsLengthRef.current`:
        - Detect whether growth was a new section or a new topic:
          - New section: find the `add-section-trigger` row index via `rows.findIndex(r => r.type === 'add-section-trigger')`; the new section row is at `triggerIdx - 1`; scroll to that index with `rowVirtualizer.scrollToIndex(triggerIdx - 1, { align: 'start', behavior: 'auto' })`
          - New topic: find the `add-topic-trigger` row whose `sectionId` matches `addTopicOpenFor` (which is still set to the open sectionId at the time the useEffect fires, before `setAddTopicOpenFor(null)` has cleared it in the next render); the new topic row is at `topicTriggerIdx - 1`; scroll to that index with `rowVirtualizer.scrollToIndex(topicTriggerIdx - 1, { align: 'start', behavior: 'auto' })`
        - Update `prevRowsLengthRef.current = rows.length` after the scroll call
      - Else: update `prevRowsLengthRef.current = rows.length` unconditionally

    Discrimination logic for section vs topic growth: a new section always increases the total rows count AND the `add-section-trigger` index increases. A new topic increases rows count AND a new `add-topic-trigger` appears for the newly opened section. Use `addTopicOpenFor` state as the discriminator — if `addTopicOpenFor !== null`, the growth was a topic addition for that sectionId; otherwise it was a section addition.

    Per CONTEXT.md decision D-02 (BUG-02): for topic additions, the parent section must already be open when `AddTopicForm` is shown (the "+ Add topic" button is only rendered inside `add-topic-trigger` rows, which only appear for non-collapsed sections per buildFlatRows line 185). So no explicit `toggleSectionOpen` call is needed in ContentTree — the section is already open when the user submits AddTopicForm.

    Anti-pattern to avoid (per RESEARCH.md Pitfall 1): do NOT call `scrollToIndex` synchronously inside AddSectionForm or AddTopicForm submit handlers. The store mutation has fired but `rows` prop has not updated yet at that point. The useEffect boundary is mandatory.

    Anti-pattern to avoid (per RESEARCH.md Pitfall 3): do NOT call `rowVirtualizer.measure()` — that forces full remeasurement of all items and is the wrong API here.

    The useEffect dependency array must include `rows` (or `rows.length`) and `rowVirtualizer` and `addTopicOpenFor`. Use `rows` as the dependency to ensure the effect fires whenever the rows array reference changes (which happens on every rows prop update from App.tsx).

    No changes to the JSX return block. No changes to AddSectionForm props. No changes to AddTopicForm props.
  </action>
