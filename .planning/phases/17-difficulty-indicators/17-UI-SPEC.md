### Difficulty Color Contract

This is the primary color addition for Phase 17. All four difficulty levels use **full-saturation -500 border** colors, matching the existing `DOT_CLASSES` in `DifficultyFilter.tsx`:

| Difficulty | Left Border Class | Badge bg (light) | Badge text (light) | Badge bg (dark) | Badge text (dark) |
|------------|-------------------|-------------------|--------------------|-----------------|-------------------|
| novice | `border-green-500` | `bg-green-100` | `text-green-700` | `dark:bg-green-900/30` | `dark:text-green-400` |
| intermediate | `border-blue-500` | `bg-blue-100` | `text-blue-700` | `dark:bg-blue-900/30` | `dark:text-blue-400` |
| advanced | `border-orange-500` | `bg-orange-100` | `text-orange-700` | `dark:bg-orange-900/30` | `dark:text-orange-400` |
| expert | `border-pink-500` | `bg-pink-100` | `text-pink-700` | `dark:bg-pink-900/30` | `dark:text-pink-400` |

Design decisions:
- Left border -500 colors: identical in light and dark mode — no `dark:` variant needed. `-500` is saturated enough for dark backgrounds.
- Badge chip colors: follow the `custom` badge pattern exactly (`bg-[color]-100 text-[color]-700 dark:bg-[color]-900/30 dark:text-[color]-400`).
- Color-to-difficulty mapping must match `DifficultyFilter.tsx` `DOT_CLASSES` exactly for cross-component visual consistency.

> Source: CONTEXT.md locked decisions "Left Border" and "Badge Chip".

### Color Map Implementation Pattern

Use static literal `Record<Difficulty, string>` objects. Do NOT use template literals or dynamic string construction — Tailwind's content scanner requires full class strings to be present in source:

```typescript
// src/components/QuestionCard.tsx (or extracted to shared constants)

const BORDER_CLASSES: Record<Difficulty, string> = {
  novice:       'border-green-500',
  intermediate: 'border-blue-500',
  advanced:     'border-orange-500',
  expert:       'border-pink-500',
};

const BADGE_BG_CLASSES: Record<Difficulty, string> = {
  novice:       'bg-green-100 text-green-700 dark:bg-green-900/30 dark:text-green-400',
  intermediate: 'bg-blue-100 text-blue-700 dark:bg-blue-900/30 dark:text-blue-400',
  advanced:     'bg-orange-100 text-orange-700 dark:bg-orange-900/30 dark:text-orange-400',
  expert:       'bg-pink-100 text-pink-700 dark:bg-pink-900/30 dark:text-pink-400',
};
```

> Source: CONTEXT.md locked decision "Use static literal color-map objects per D-06". Same pattern as `DOT_CLASSES` in `DifficultyFilter.tsx`.

### DOM Structure

Apply the left border to the **outer QuestionCard container `<div>`**, not a sub-element:

```tsx
// Outer div: add border-l-4 + dynamic border color
<div className={`bg-white dark:bg-gray-900 border-b border-gray-100 dark:border-gray-800 border-l-4 ${BORDER_CLASSES[question.level]}`}>
```

The badge chip `<span>` goes after the question text `<span>` and before the note icon button, within the flex row:

```tsx
{/* Difficulty badge chip */}
<span className={`text-xs font-normal px-1.5 py-0.5 rounded uppercase shrink-0 ${BADGE_BG_CLASSES[question.level]}`}>
  {question.level}
</span>
```

> `shrink-0` prevents badge from collapsing in flex row — replicates `custom` badge pattern (line 85, `QuestionCard.tsx`).
> `uppercase` CSS utility produces the required `NOVICE` / `INTERMEDIATE` etc. display without string manipulation.
> Source: CONTEXT.md locked decisions "Badge Chip — position" and "shrink-0 on badge chips" pattern.

## States Required

### QuestionCard Left Border

| State | Visual |
|-------|--------|
| novice question | `border-l-4 border-green-500` on outer container |
| intermediate question | `border-l-4 border-blue-500` on outer container |
| advanced question | `border-l-4 border-orange-500` on outer container |
| expert question | `border-l-4 border-pink-500` on outer container |
| dark mode | Same border colors — no dark: variant (saturated -500 works on dark backgrounds) |
| print | Border visible — no `print:hidden` (difficulty context useful in print output) |
| custom question | Same border applied — custom questions have a `level` field and indicators should be consistent |

### Difficulty Badge Chip

| State | Visual |
|-------|--------|
| novice | `bg-green-100 text-green-700 dark:bg-green-900/30 dark:text-green-400` — text renders "NOVICE" |
| intermediate | `bg-blue-100 text-blue-700 dark:bg-blue-900/30 dark:text-blue-400` — text renders "INTERMEDIATE" |
| advanced | `bg-orange-100 text-orange-700 dark:bg-orange-900/30 dark:text-orange-400` — text renders "ADVANCED" |
| expert | `bg-pink-100 text-pink-700 dark:bg-pink-900/30 dark:text-pink-400` — text renders "EXPERT" |
| always visible | Not a hover state — badge is always rendered, difficulty is a primary informational signal |
| print | Visible in print — no `print:hidden` |

### Print Row (existing print-only div)

The print-only `<div>` (currently shows question text + score) should also include the difficulty badge chip so printed output retains difficulty context. Apply the same badge `<span>` markup within the print row.

## Accessibility Requirements

- Badge chip `<span>` must include `aria-label="{level} difficulty"` (e.g. `aria-label="novice difficulty"`) so screen readers announce the full label rather than the uppercase visual text.
- Left border is purely decorative — no `aria-*` attribute required on the border element.
- Existing `focus-visible:ring-2 focus-visible:ring-blue-500 focus-visible:outline-none` pattern on interactive controls is unchanged — no regression.
- Color is not the only means of conveying difficulty (the text badge chip also communicates the level) — satisfies WCAG 1.4.1 Use of Color.
- All badge chip color combinations pass WCAG 2.1 AA for normal text at 12px (see contrast table in Color section).
